单例模式

发表于 2017-01-25

单例模式简介

定义：确保一个类在内存中只有一个实例，而且自行实例化，并向整个系统提供这个实例。单例模式是应用最广的模式之一，同时也是最简单的一种设计模式，一般如果系统中创建某个对象比较消耗资源，且只有一个实例即可，这样使用单例是最合适的。

单例模式使用场景

常见某个对象比较消耗资源，或者要确保某个类有且只有一个实例，则要考虑使用单例模式。例如：创建某个对象，需要访问IO或者数据库资源等，则考虑使用单例模式。

单例模式UML


Client类为客户端，Singleton类为单例类，单例中一般要保证几点：

  1. 构造方法私有化private；
  2. 对外暴露静态方法获取单例对象；
  3. 保证在多线程中也能保证只有一个单例对象；
  4. 在反序列化中也能保证不会重新创建对象；

以上几点是单例中的几个关键点，主要关键点是在多线程中要保证线程安全，确保对象只有一个。

这种实现方式，通过双重判空，可以避免每次调用getInstance的时候进行不必要的同步，第一层判空来避免不必要的同步，第二层判断来进行null的状态下创建对象，对于复杂的创建对象的场景，第一次创建对象时，反应稍慢，对象创建成功后就不需要再进行同步了。但是在java1.5之前的Java内存模型中 ，创建对象的操作mInstance = new Instance()会进行一下过程：

  1. 给Singleton分配实例
  2. 调用SingleInstance()构造方法，初始化成员变量
  3. 将mInstance指向对象分配的内存空间(此时mInstance就不为空了)

由于Java编译器是允许处理器乱序执行的，所以当线程A执行mInstance = new Singleton()时，可能会以1-3-2的顺序执行，如果执行到3以后，这是刚好cpu切换到了线程B，这是由于mInstance已经不为空了，所以就将mInstance取走了，但是此时并未初始化完成，所以造成了DCL失效问题，但是除非并发场景很复杂或者使用的JDK低于JDK6，否则这种方式一般是可以满足需求的。是一种常用的方式。
